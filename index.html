🧪 DINÁMICA 1 — Elaboración de Test Cases

Cumple con los 5 puntos de desarrollo solicitados, con ejemplos claros y visuales.

📚 Desarrollo — 5 apartados (UX mejorado)
1

Repaso teórico – ¿Qué es un Test Case?

📌 Un Test Case es un conjunto de precondiciones, entradas, pasos, resultado esperado y criterios de aceptación para validar un requisito.
  • ID: TC‑RF‑01
  • Precondiciones: estado del sistema necesario (p. ej., “usuario no registrado”).
  • Entradas/Datos: valores de prueba (válidos/ inválidos, límites).
  • Pasos: instrucciones reproducibles (1…n).
  • Resultado esperado: salida observable y verificable.
  • Criterios de aceptación: métrica/umbral para aprobar (p. ej., “P95 < 1.2 s”).
2

Ejemplo — Buen vs Mal Caso de Prueba

✅ Buen caso

Objetivo: Consultar calidad del aire con apiKey válida.

  • Precondición: apiKey válida registrada.
  • Datos: ciudad=SCZ, fecha=hoy.
  • Pasos: GET /air?city=SCZ.
  • Esperado: HTTP 200 en < 3 s con payload ICA.
  • Criterio: t_resp P95 < 3000 ms; esquema JSON válido.

❌ Mal caso

“Probar API rápido”. Sin datos, sin pasos, sin métrica ⇒ no es verificable.

3

Módulo asignado — Plataforma Ambiental

  • Ámbitos: Ingesta APIs, ETL, Modelos ML, Automatizaciones n8n, Dashboards, App.
  • Actores: Industria, Gobierno, Ciudadanía.
  • Resultados: alertas, reportes PDF, tendencias ICA.
🎯 Objetivo del módulo: transformar datos urbano‑industriales en predicciones y acciones (alertas/reportes).
4

Identificación de requisitos — 5 RF y 3 RNF

  • RF1: Registrar usuario (industria/autoridad/ciudadano)
  • RF2: Consultar calidad del aire por API
  • RF3: Predicción de contaminación (series temporales)
  • RF4: Generación de reporte PDF semanal
  • RF5: Clasificación de quejas ciudadanas (reglas)
  • RNF1: Rendimiento API (P95 por carga)
  • RNF2: Compatibilidad web (Chrome/Firefox/Edge/Safari)
  • RNF3: Seguridad login (prevención de SQLi)

Estos requisitos ya están ligados a los casos del repositorio (más abajo).

5

Técnicas de diseño aplicadas (2+)

Partición de Equivalencias

  • Idea: agrupar entradas en clases que se comportan igual.
  • Uso aquí: API con clave válida vs inválida; navegadores soporte vs no.
  • Beneficio: menos pruebas, misma cobertura.

Valores Límite

  • Idea: probar en los bordes donde suelen fallar los sistemas.
  • Uso aquí: umbrales de ICA (0, 35, 36, 150, 151) y carga 50/100/200 rqm.
  • Beneficio: detecta off‑by‑one, saturaciones y cortes.

Tablas de Decisión

  • Idea: reglas condición → acción para enrutar quejas y alertas.
  • Uso aquí: (tipo×gravedad) → responsable; (umbral×zona) → notificación.
  • Beneficio: cobertura combinatoria sin olvidar casos.

Métricas de aceptación

  • Funcional: 100% de válidos aceptados / inválidos rechazados.
  • Rendimiento: P95 respuesta < 0.8/1.2/2.0 s (50/100/200 rqm).
  • Usabilidad: SUS ≥ 80/100 en dashboards.
  • Seguridad: 100% bloqueo de SQLi/inputs maliciosos.

🛠️ Herramientas (TestLink vs Squash)
  • Squash TM: trazabilidad RF→TC→Ejecuciones, etiquetas, REST API, CI fácil.
  • TestLink: clásico; requiere plugins para integraciones modernas.

📋 DINÁMICA 2 — Test Plan ISO 9001:2015

Plan editable + cronograma visual tipo Trello.

📚 Desarrollo

Objetivo de pruebas

  • Garantizar la provisión consistente de productos y servicios.
  • Aumentar la satisfacción del cliente mediante un SGC con enfoque a procesos, ciclo PHVA y pensamiento basado en riesgos.
  • Validar que la ingesta de datos, modelos predictivos, automatizaciones n8n, APIs y dashboards cumplen requisitos funcionales y no funcionales.
  • Generar evidencias documentadas y promover la mejora continua.

Alcance

  • Incluye: Ingesta de datos (APIs públicas y reportes), ETL → PostgreSQL/MongoDB, modelos predictivos (entrenamiento y serving), automatizaciones n8n (alertas/reportes), APIs de consulta/predicción, dashboards (Metabase/Power BI) y app ciudadana, generación de PDF y gestión de quejas.
  • Interacciones con proveedores externos de datos.
  • Excluye: hardware IoT propietario y etiquetado manual de datos.

Estrategia y niveles (PDCA)

  • Planificar (Cl. 6.1, 6.2): objetivos de calidad medibles, riesgos y oportunidades, criterios de aceptación, recursos y calendario.
  • Hacer (Cl. 8.1, 8.3, 8.5.6): pruebas unitarias, integración, sistema y aceptación; tipos funcional, rendimiento, seguridad y usabilidad; control de diseño y cambios; operación controlada.
  • Verificar (Cl. 9.1, 9.1.2): seguimiento y medición de métricas; validación de criterios; satisfacción del cliente; análisis de datos.
  • Actuar (Cl. 10.2, 9.3): acciones correctivas; actualización de riesgos; revisión por la dirección.
  • Proveedores externos (Cl. 8.4) e información documentada (Cl. 7.5): evaluación/ control y trazabilidad con control de versiones.

Recursos (humanos/técnicos)

  • Personas: QA Lead, Backend, Frontend, Data/ML, DevOps (competencia demostrable; Cl. 7.2).
  • Infraestructura: repositorios Git con control de versiones, CI/CD (GitHub Actions), ambientes DEV/QA/PROD, Postman, JMeter, Cypress, Sonar; orquestación de jobs; almacenamiento de evidencias.
  • Comunicación: qué/cuándo/a quién/cómo (Cl. 7.4). Seguimiento/medición: recursos adecuados y mantenidos (Cl. 7.1, 7.1.5).

Criterios de entrada/salida

  • Entrada: alcance y requisitos priorizados; objetivos de calidad; datos de prueba; ambientes listos; responsables y comunicación definidos; control de versiones (Cl. 7.5).
  • Salida (aceptación): conformidad con criterios por módulo/actor; P95 bajo umbrales; 0 vulnerabilidades críticas; SUS ≥ 80; evidencias y trazabilidad completas; registro de no conformidades y acciones correctivas (Cl. 8.7, 10.2); reporte para revisión por la dirección (Cl. 9.3).
